# UI & gRPC-Gateway

This document describes the design and implementation plan for the HTTP gateway and web UI that sit in front of the payload-manager gRPC API.

## Overview

Two new components are added to the repo:

- **`gateway/`** — a Go [grpc-gateway](https://github.com/grpc-ecosystem/grpc-gateway) binary that translates REST/JSON HTTP requests into gRPC calls and serves the compiled UI
- **`ui/`** — a [Svelte](https://svelte.dev) SPA for browsing payloads, viewing metadata and lineage, and performing actions (spill, pin, promote, delete)

Neither component modifies any existing C++ source, proto, or CMake files.

## Architecture

### Code generation pipeline

```
buf generate
  → gateway/gen/go/...          Go gRPC client stubs
  → gateway/openapi/...         OpenAPI 2.0 spec

orval
  → ui/src/lib/client/index.ts  Typed TypeScript fetch client
```

The gateway HTTP mappings live in `gateway/api_config.yaml` — a `google.api.Service` YAML file consumed by the grpc-gateway protoc plugin. This means **no `google/api/annotations.proto` imports are added to the proto files** and the C++ build is unaffected.

Buf managed mode sets `option go_package` automatically via `go_package_prefix` in `buf.gen.yaml`, so no proto files need to be touched.

### UUID handling

Payload and lease IDs are 16-byte binary UUIDs (`PayloadID.value`, `LeaseID.value`). grpc-gateway expects bytes path parameters to be base64url-encoded. `gateway/middleware/uuid.go` rewrites any standard UUID hex string (`xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx`) in the URL path to `base64.RawURLEncoding` before routing, so the UI and API clients can use normal UUID strings in URLs.

`ui/src/lib/uuid.js` provides `uuidToBase64()` and `base64ToUuid()` helpers for the frontend.

### Live data

The payload list and stats bar poll every 5 seconds via Svelte stores. The streams page uses `GET /v1/streams/{ns}/{name}/subscribe` (a grpc-gateway server-streaming endpoint that returns newline-delimited JSON chunks) for a live feed of new stream entries.

### Single binary deployment

`gateway/main.go` embeds the compiled UI via `//go:embed static`. The build process copies `ui/dist/` to `gateway/static/` before `go build`. The resulting binary serves both the REST API at `/v1/` and the SPA at `/`.

| Method | Path | RPC |
|--------|------|-----|

## UI pages

### Payloads

- Tier filter and state filter dropdowns
- Table: ID, tier badge, state, size, age, active leases
- Click any row → slide-in detail panel
- **Detail panel** — three tabs:
  - **Descriptor** — placement info (tier, location, version, timestamps)
  - **Metadata** — JSON view + update editor
  - **Lineage** — upstream/downstream edges with clickable UUID links
- **Action buttons** — Spill, Promote, Pin, Unpin, Prefetch, Delete (with confirm)
- **Allocate** button to create new payloads

### Streams

- List of known streams (persisted in `localStorage`)
- Create stream modal
- Stream browser: offset-paginated entry table with live toggle

### Admin

- Per-tier stat cards (payload count + bytes) auto-refreshing every 5s

## Building

### Prerequisites

- [buf](https://buf.build/docs/installation) — proto toolchain
- Go 1.22+
- Node 20+

### Commands

```bash
# generate all code (proto stubs, OpenAPI spec, TS client)
make generate

# build gateway binary with embedded UI
make build

# run locally against a gRPC server
cd gateway && ./bin/gateway --grpc-addr localhost:50051 --http-addr :8080
```

The gateway also reads `GRPC_ADDR` and `HTTP_ADDR` environment variables as fallback for the CLI flags.

## Deployment

The gateway is packaged as a single statically-linked binary with the UI embedded. It is added to the existing Docker Compose stack as an overlay file.

### Docker

```bash
# generate proto stubs before building (required — Dockerfile does not run buf)
buf generate

# build images
docker compose \
  -f docker/docker-compose.sqlite.yml \
  -f docker/docker-compose.gateway.yml \
  build

# run
docker compose \
  -f docker/docker-compose.sqlite.yml \
  -f docker/docker-compose.gateway.yml \
  up
```

UI available at `http://localhost:8080` once the gateway health check passes.

### Compose overlay combinations

```bash
# SQLite (default)
docker compose -f docker/docker-compose.sqlite.yml \
               -f docker/docker-compose.gateway.yml up

# Postgres
docker compose -f docker/docker-compose.postgres.yml \
               -f docker/docker-compose.gateway.yml up

# Postgres + OTEL
docker compose -f docker/docker-compose.postgres.yml \
               -f docker/docker-compose.otel.postgres.yml \
               -f docker/docker-compose.gateway.yml up

# GPU + Postgres
docker compose -f docker/docker-compose.gpu.postgres.yml \
               -f docker/docker-compose.gateway.yml up
```

### What the gateway container does NOT do

- It does not expose the gRPC port — it connects to `payload-manager:50051` on the internal compose network
- It does not run `buf generate` at image build time
- It does not require any database access
